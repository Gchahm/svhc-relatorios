# Phase 1 Data Model: Derive the classification plan from D1

## Schema change

### `attachments` (table) — add one column

| Column | Type | Null | Notes |
|--------|------|------|-------|
| `content_hash` | TEXT | YES (nullable) | Stable byte-content fingerprint of the attachment's page images. The grouping key for shared-NF detection. NULL when pages are absent/unreadable or for data captured before this feature. |

- Drizzle: `contentHash: text("content_hash")` on `attachmentsTable` in `src/db/fiscal.schema.ts`.
- Migration: generated by `pnpm db:generate` → `drizzle/0008_*.sql` (`ALTER TABLE attachments ADD COLUMN content_hash text;`). Committed.
- No index required (group set is small, ~hundreds of rows per period; grouping is done in Python after a per-period `SELECT *`).
- The Python upsert path (`common/d1.py:_generate_inserts`) derives columns from row keys, so adding
  `"content_hash"` to the scraper's attachment dict makes it persist with no SQL-generator change.

**Value semantics** (unchanged from `nf_groups.content_hash`):
- Joined md5 over each page file's bytes, each page length-delimited with `:{size}:`, in page order.
- `None`/NULL when there are no pages or any page is unreadable → the attachment is ungroupable
  (singleton).

## Derived structures (in-memory only — no persistence)

### `GroupPlan` (replaces a manifest `groups[]` entry)

A pure-Python value built by `build_plan(...)` from D1 rows + materialized images. Not stored.

| Field | Meaning |
|-------|---------|
| `group_key` | the content hash (or `doc:{id}` for an ungroupable singleton) |
| `representative_attachment_id` | id of the highest-amount member (items sorted amount-desc) |
| `group_size` | number of attachments in the full group (pre-filter) |
| `sibling_sum` | sum of member entry amounts over the full group (pre-filter) |
| `pages[]` | representative's pages: `{page_index, page_label, path, read_path}` (read_path = absolute materialized cache path) |
| `members[]` | per-member context: `{attachment_id, entry_id, entry_amount, vendor_name, is_representative}` |

This is structurally identical to the previous manifest's `groups[]`, so neither the
`classify-period` consumer nor the `apply-extractions` logic changes shape — only the *source*
(a function/stdout) changes from a file.

## Relationships & invariants

- `attachments.content_hash` groups attachments 1:N within a period (one hash → many sibling
  attachments). Cross-period grouping is not done (grouping is computed per loaded period set, as today).
- **Invariant (FR-004)**: a NULL/empty hash never merges distinct attachments — it always yields a
  singleton keyed by the attachment id.
- **Invariant (FR-002)**: byte-identical page sets ⇒ equal `content_hash` ⇒ same group, on both
  freshly-captured (column) and legacy (fallback) data.
- **Backfill**: `materialize_period_images` may UPDATE `content_hash` from NULL → computed value; it
  never overwrites a non-NULL value and never changes any other column.

## Read paths (unchanged)

- `loader._load_period_raw` already does `SELECT d.* FROM attachments d ...`, so `content_hash` is
  carried into the in-memory `attachments[]` automatically once the column exists.
- The frontend (`src/...`) does not read `content_hash`; no UI change.
