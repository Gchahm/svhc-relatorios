# Implementation Plan: Atomic delete-then-insert writebacks (no partial-failure data loss)

**Branch**: `024-atomic-writebacks` | **Date**: 2026-06-11 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/024-atomic-writebacks/spec.md` (GitHub issue #37, BUG-006)

## Summary

Two analysis-pipeline writebacks run "clear old rows" and "insert new rows" as **two separate
`wrangler` invocations**, so a failure between them commits the delete but not the insert and
silently loses data (attachment analyses; period/overpayment alerts). The attachment path is worse:
the same first call also stamps `attachments.classified_at`, so a half-write leaves the attachment
marked done with no analysis and it never gets re-attempted.

The fix folds each writeback's `DELETE`(s) and `INSERT`(s) — and, for attachments, the
`classified_at` stamp — into a **single `d1.execute_sql` batch**. D1 runs a submitted file's
statements in one implicit transaction, so a failure anywhere in the batch rolls back the whole
writeback: the prior rows survive, the attachment stays pending, and the next run heals it. A new
`d1.upsert_sql(...)` helper exposes the INSERT-generation already inside `upsert_tables` as a string
(without executing it), so callers can prepend their `DELETE`/`UPDATE` and submit one file. Errors
already propagate from `execute_sql` (non-zero `wrangler` exit → `CalledProcessError`); the change
keeps that loud-failure behavior.

## Technical Context

**Language/Version**: Python 3.12+ (analysis CLI under `scripts/`, run via `uv`); stdlib only.
**Primary Dependencies**: `scripts/common/d1.py` (the `wrangler`-CLI wrapper) — `execute_sql`,
`upsert_tables`, `build_sql`, `query`. No new pip/npm dependencies.
**Storage**: Cloudflare D1 (SQLite) via the `wrangler` CLI. Tables touched: `attachment_analyses`,
`attachment_analysis_records`, `attachments` (the `classified_at` stamp — analysis-owned column),
`alerts`. **No schema migration** (write-sequencing fix only; FR-008).
**Testing**: Python stdlib `unittest` (no test framework is configured and the constitution forbids
mandating one the repo can't run; `unittest` runs via `python -m unittest` with zero new deps). Tests
intercept the `d1.execute_sql` / `d1.upsert_tables` seam (monkeypatch) to assert: (a) each writeback
submits **one** combined batch; (b) the `classified_at` `UPDATE` is in the same batch as / after the
analysis `INSERT`, never alone with the `DELETE`; (c) a simulated insert failure raises and performs
no separate committed delete.
**Target Platform**: Local dev (Miniflare) and Cloudflare D1 production (`--remote`); the Python CLI
runs on Linux/macOS.
**Project Type**: Single project — Python analysis CLI (the Next.js frontend is unaffected; it only
reads these tables).
**Performance Goals**: No regression. Folding two `wrangler` calls into one per writeback is a net
reduction in process spawns (slightly faster), not slower.
**Constraints**: stdlib only; no schema change; preserve byte-for-byte the rows written today
(parity, FR-006) and the alert resolution/notes preservation (FR-007).
**Scale/Scope**: Two writeback paths; ~3 functions edited (`attachments._merge_and_write`,
`__init__.run_analysis` per-period + overpayment writebacks) plus one new helper in `common/d1.py`.

  constitution's "tests added when a spec requests them; must pass before merge."
- **IV. Security & Auth by Default** — PASS. No routes, auth, or secrets touched; SQL is generated by
  the existing escaped `build_sql`/`_escape_sql` path (no new injection surface — ids are already
  `.replace("'","''")`-escaped as today).
- **V. Simplicity & Incremental Delivery** — PASS. The simplest fix (concatenate the existing
  DELETE/UPDATE SQL with the existing INSERT SQL into one `execute_sql` call) — no new abstraction
  beyond a thin string-returning twin of `upsert_tables`. No transaction framework, no new dependency.

functions and one new helper in the shared D1 wrapper. Tests live under `scripts/tests/` (new), run
with `python -m unittest discover -s scripts/tests` (or `uv run`), importing the analysis/common
modules with `d1`'s subprocess calls monkeypatched so no real `wrangler`/network is needed.
