# SVHC Relatórios Constitution

Fiscal auditing tool for condominium SVHC — a Next.js 15 application deployed to Cloudflare
Workers via OpenNext, backed by Cloudflare D1 (SQLite) through Drizzle ORM.

## Core Principles

### I. Type Safety & Schema Discipline

TypeScript is the contract. Code MUST type-check cleanly; `any` is permitted only with an
inline justification comment. The Drizzle schema is the single source of truth for persisted
data — schema changes MUST flow through `pnpm db:generate` and a committed migration in
`drizzle/`, never through ad-hoc SQL or hand-edited migration files. `src/db/auth.schema.ts`
is generated by the better-auth CLI and MUST NOT be edited by hand; after any auth config
change, run `pnpm auth:update` and commit the regenerated schema.

**Rationale**: A typed, generated schema prevents drift between code, database, and auth state,
which is the most common source of silent data corruption in this stack.

### II. Cloudflare-Native Architecture

All database access MUST go through `await getDb()` from `@/db`, which resolves the D1 binding
from the Cloudflare context — never construct connections directly. Code that depends on
runtime bindings MUST NOT assume `getCloudflareContext()` is available during build; keep
build-time and runtime paths separate (as the auth module does for CLI schema generation).
Bindings and environment variables are declared in `wrangler.toml`; new external dependencies
MUST be expressed as Cloudflare bindings rather than hardcoded endpoints or credentials.

**Rationale**: The Workers/D1 runtime has hard edge constraints; honoring them at design time
avoids deploy-only failures that cannot be reproduced in a plain Node environment.

### III. Quality Gates Before Commit

Every commit MUST pass `pnpm lint` and `pnpm format` (Prettier: 4-space tabs, 120 print width,
double quotes, ES5 trailing commas). Formatting and lint are non-negotiable gates, not
suggestions. Automated tests are OPTIONAL — the repository has no test framework configured, so
tests are added only when a feature spec explicitly requests them; when present they MUST pass
before merge. Manual verification of the running app SHOULD accompany any user-facing change.

**Rationale**: Consistent formatting and a clean lint pass keep diffs reviewable and reduce
churn; mandating tests the project cannot currently run would be aspirational fiction rather than
an enforceable rule.

### IV. Security & Auth by Default

Authentication is handled by better-auth; new routes that expose user or fiscal data are
protected by default and MUST justify any public exposure explicitly. The middleware session
check pattern (fetching `/api/auth/get-session` internally) MUST be preserved to avoid Edge
Runtime eval issues. Secrets and credentials MUST live in Cloudflare bindings/environment
configuration, never in source or client-shipped code.

**Rationale**: This is a fiscal auditing tool handling sensitive condominium financial data;
defaulting to protected and keeping secrets out of the bundle is the floor, not an enhancement.

### V. Simplicity & Incremental Delivery

Prefer the simplest solution that satisfies the requirement (YAGNI). New abstractions,
dependencies, or projects MUST be justified against a concrete present need, not a hypothetical
future one. Features are delivered as independently testable, independently shippable increments
ordered by priority, consistent with the spec-driven workflow (specify → plan → tasks →
implement). Reuse existing shadcn/ui components and the `cn()` utility before introducing new UI
primitives.

**Rationale**: A small team and an edge runtime reward minimal surface area; incremental slices
keep each change demonstrable and reversible.

## Technology Constraints

- **Framework**: Next.js 15 (App Router, React 19). New pages and APIs follow the App Router
  conventions already established under `src/app/`.
- **Deployment**: Cloudflare Workers via `@opennextjs/cloudflare`. `pnpm deploy` is the only
  supported deploy path.
- **Database**: Cloudflare D1 (SQLite) via Drizzle ORM. Migrations applied with
  `pnpm db:migrate:dev` (local) and `pnpm db:migrate:prod` (production).
- **UI**: shadcn/ui (new-york style), Tailwind CSS 3, Radix primitives, lucide-react icons.
- **Package manager**: pnpm. Lockfile changes MUST be committed alongside dependency changes.
- **Path alias**: `@/*` maps to `./src/*`; use it instead of long relative paths.

## Development Workflow

- Features follow the speckit pipeline: `specify` → `clarify` (as needed) → `plan` → `tasks` →
  `analyze` (as needed) → `implement` → `pr`. Specs live under `specs/<NNN>-<short-name>/`.
- Branches are named `<NNN>-<short-name>` per `.specifyrc`; work happens on a feature branch,
  not directly on `main`.
- Before opening a PR: run `pnpm lint` and `pnpm format`, and confirm migrations and any
  generated schema files are committed.
- The `plan` and `analyze` phases MUST validate proposed work against the Core Principles above
  and record any deviation in the plan's Complexity Tracking section.

## Governance

This constitution supersedes ad-hoc practice. When a principle and convenience conflict, the
principle wins or the principle is amended — it is not silently ignored.

- **Amendments**: Proposed via PR editing this file, with a Sync Impact Report (see top comment)
  describing the version bump and any dependent template/doc updates. Amendments take effect when
  merged to `main`.
- **Versioning**: Semantic. MAJOR for backward-incompatible governance/principle removals or
  redefinitions; MINOR for a new principle or materially expanded guidance; PATCH for
  clarifications and wording fixes.
- **Compliance**: PR review MUST verify changes comply with the Core Principles. Any deviation
  MUST be justified in the relevant plan's Complexity Tracking table or rejected.
- **Runtime guidance**: Day-to-day implementation detail lives in `CLAUDE.md` and `AGENTS.md`;
  those documents MUST stay consistent with this constitution and defer to it on conflicts.
