# CLAUDE.md

## Project Overview

Next.js 15 app deployed to Cloudflare Workers via OpenNext. Fiscal auditing tool for condominium SVHC.

## Tech Stack

- **Framework:** Next.js 15 (App Router, React 19)
- **Deployment:** Cloudflare Workers via `@opennextjs/cloudflare`
- **Database:** Cloudflare D1 (SQLite) with Drizzle ORM
- **Auth:** better-auth with Cloudflare adapter (openAPI plugin)
- **UI:** shadcn/ui (new-york style), Tailwind CSS 3, Radix primitives, lucide-react icons
- **Package manager:** pnpm
- **Formatting:** Prettier (4-space tabs, 120 print width, double quotes, trailing commas es5)

## Key Commands

```bash
pnpm dev              # Local dev server
pnpm build            # Next.js build
pnpm deploy           # Build + deploy to Cloudflare
pnpm preview          # Build + local Cloudflare preview

# Database
pnpm db:generate      # Generate Drizzle migrations
pnpm db:migrate:dev   # Apply migrations locally
pnpm db:migrate:prod  # Apply migrations to production
pnpm db:studio:dev    # Drizzle Studio (local)

# Auth schema
pnpm auth:update      # Regenerate auth schema + format

# Formatting
pnpm format           # Prettier format all files
```

## Important Patterns

- **DB access:** Always use `await getDb()` from `@/db` - it retrieves the D1 binding from Cloudflare context
- **Auth schema:** `src/db/auth.schema.ts` is auto-generated. Run `pnpm auth:update` after changing auth config
- **Cloudflare context:** `getCloudflareContext()` is only available at runtime, not during build. The auth module has a separate static config for CLI schema generation
- **Middleware:** Checks auth by fetching `/api/auth/get-session` internally to avoid Edge Runtime eval issues
- **Path alias:** `@/*` maps to `./src/*`
- **Environment:** Cloudflare bindings defined in `wrangler.toml`: `DATABASE` (D1), `KV`, `DOCUMENTS` (R2), `ASSETS`; env vars: `BETTER_AUTH_URL`, `BETTER_AUTH_TRUSTED_ORIGINS`
- **Attachments vs. Documents (terminology — feature 015):** An **attachment** (table `attachments`, one per entry — the renamed former `documents`) is the multi-page bundle downloaded from the brcondos portal; its analysis chain is `attachment_analyses` (roll-up, FK `attachment_id`) + `attachment_analysis_records` (per-page, FK `attachment_analysis_id`). The word **document** is **reserved** for a _real_ fiscal document (an NF/receipt/boleto identified within attachment pages) — a future entity that will be **N:N with entries** (e.g. "NF #X linked to 4 entries; sum of entries ≤ NF value"). That entity does **not** exist yet. KEEP exceptions where "document" is intentional: the portal id `external_document_id`, the real-doc fields `document_type`/`document_number` and VLM `tipo_documento`, the R2 binding `DOCUMENTS`/bucket `fiscal-documents`/`getDocumentsBucket()` (storage infra), the portal `/ViewDocuments/` URL + scraper `extractors/documentos.py`, and the `--download-docs` flag.
- **Attachment page images (R2):** Page images live in the `DOCUMENTS` R2 bucket (`fiscal-documents`), reached via `getDocumentsBucket()` in `@/lib/r2` (mirrors `getDb()`). The object key is the path **relative to `data/scrape/`** — `<period>/<basename>` (e.g. `2025-12/<entry_id>_p1.png`) — derived from `attachments.file_path` by `objectKeyFromFilePath()`; there is **no D1 column** for it. Page images are uploaded to R2 **during the scrape run** (`python -m scraper scrape --download-docs [--remote]`, via `getDocumentsBucket`-equivalent `wrangler` calls in `scripts/common/d1.py`); the scraper writes `attachments.file_path` as the `;`-joined `<period>/<basename>` keys it uploaded (keep that form in sync with `@/lib/r2`). For analysis, those images are materialized back from R2 into the ephemeral local cache `.cache/analysis/<period>/` (`scripts/analysis/images.py`) so the vision skill (and the legacy NF content-hash fallback/backfill) have local files. The detail dialog (`AttachmentAnalysisDetailDialog`) renders each page via `PageImageViewer` next to its extracted record; the auth-gated route `GET /api/attachment-analyses/[id]/image/[page]` resolves the key server-side (the client never sends a raw key) and streams the bytes, while `GET /api/attachment-analyses/[id]/pages` lists the pages. Locally Miniflare fabricates the bucket; in prod run `wrangler r2 bucket create fiscal-documents` once.
- **Pre-commit:** Always run `pnpm lint` and `pnpm format` before committing to ensure code passes ESLint and Prettier
- **Direct-to-D1 scrape & analysis (feature 014):** the scraper and analysis pipeline read/write **Cloudflare D1 + R2 directly** — there is no `data/scrape/<period>.json` and no separate import step. `python -m scraper scrape --download-docs [--remote]` upserts a period's ledger rows into D1 and uploads its page images to R2 in one run (default **local**; `--remote` for production). The analysis CLI (`python -m analysis …`) takes the same `--remote` flag, loads periods from D1 (`scripts/analysis/loader.py`), and materializes images from R2 into `.cache/analysis/` (`scripts/analysis/images.py`). All D1/R2 access from Python goes through `scripts/common/d1.py` (a thin `wrangler`-CLI wrapper: `query`/`upsert_tables`/`execute_sql`/`put_object`/`get_object`), which also owns the SQL generation (escaping, `TABLE_ORDER`, `INSERT OR REPLACE`, `analysis_records` flattening) formerly in the retired `import-to-d1.mjs`/`upload-images-to-r2.mjs`.
- **Attachment analysis (Claude vision skills):** the VLM (`mlx_vlm`) flow is retired. The flow is: (1) the **`classify-period` skill** runs `docs-plan` itself for a period, which **derives the work plan from D1 and prints it as JSON to stdout** (no manifest file — the shared-NF grouping comes from `attachments.content_hash`; feature 016 — and each plan page carries a `recorded` flag, true when its extraction is already in D1; feature 017), then fans each representative page out to (2) the **`classify-doc-page` skill**, which views one page image (a materialized cache file) and **records that page's frozen-field extraction directly to D1** via `python -m analysis record-classification --attachment-id <id> --page <label> [--remote]` (one row per page in the `page_classifications` staging table — there is **no** `.classify.json` file; feature 017). The skill is passed each page's `attachment_id` + `page_label` by the orchestrator (page-image files are named by *entry*, not attachment); the `record-classification` CLI is the contract-validation point (the former PostToolUse file-write hook is gone). (3) `python -m analysis apply-extractions --periodo <p> [--remote]` reads each page's recorded extraction **from D1** (via `D1ExtractionProvider`, sourced from the loader's `page_classifications` rows) and writes the rolled-up authoritative `attachment_analyses` (+ flattened `attachment_analysis_records`) to D1 via delete-then-insert, then stamps `attachments.classified_at` — so clearing the cache between classify and apply no longer loses vision work; (3b) **work selection is DB-controlled** — the plan is the _pending_ set (`attachments.classified_at IS NULL`); to (re)classify a subset, mark it pending in D1 with `python -m analysis mark-pending --attachment-id <ids…>` (clears `classified_at`) rather than threading id flags through the CLI/skills (feature 016); (4) `analyze` writes alerts to D1 (delete-then-insert per `reference_period`) and `mismatches` prints the terse summary. The **`analyze-docs` agent** is the context-isolated wrapper that invokes `classify-period` then runs apply/analyze/mismatches and returns only the summary. Each page yields a `page_extraction` record nested as `analysis_records` under its `attachment_analyses` object; `scripts/common/d1.py:upsert_tables` flattens these into `attachment_analysis_records` on write. The `attachment_analyses` row is a heterogeneity-aware roll-up (amount validated against paid/net when a payment artifact is present); per-artifact values (gross/net/paid) live in the per-page records. The extraction-source seam is `attachments.build_attachment_analysis(..., provider)` (provider keyed by `(attachment_id, page_label)`); the staging-table seam (validate / record / read) is `scripts/analysis/page_classifications.py`; the deterministic plan/apply/summary live in `scripts/analysis/extractions.py`. The legacy `attachment_analyses.raw_response` / `attachment_analysis_records.raw_text` columns were dropped (feature 017; migration `0010`) — they had been unused since the VLM path was retired.
- **Shared-NF grouping & reconciliation:** The source system attaches one Nota Fiscal to several entries (line-item splits, or principal vs. `JUROS/MULTAS`). Attachments are grouped by **byte-identical page content** via the persisted `attachments.content_hash` (`scripts/analysis/nf_groups.py:group_attachments` reads that column, written at scrape time from the page bytes by `scripts/common/hashing.py:content_hash`, with a compute-from-cache fallback + lazy backfill for rows captured before the column existed — `file_path` and `external_document_id` differ per sibling, and the extracted NF number is noisy, so the content hash is the authoritative key). For a multi-entry group, the extraction runs **once** per unique NF (the DB-derived plan lists only the representative's pages, `classify-doc-page` views them, and `apply-extractions` fans the extraction out to siblings), then validates `amount_match` by reconciling **`sum(sibling amounts)` against the NF gross total** (`nf_total_for_reconciliation` prefers the invoice `valor_total`, else the roll-up), not the full total against each fractional entry — this removes the false mismatches on splits. Single-entry attachments keep the original per-entry comparison. Tolerance is reused (`nf_groups.reconcile_group`: within 5% relative OR R$ 0.05 absolute → `reconciled`; sum > total → `over_claim`; sum < total → `under_claim`).
- **Duplicate-billing alert:** `check_duplicate_billing` (`scripts/analysis/checks/advanced.py`, run by `analyze`/`run_all_checks`) emits a `critical` alert of type `duplicate_billing` when a shared NF's sibling entries sum to **more** than the NF total (an over-claim — the same NF claimed above its face value), distinct from a legitimate split (`reconciled`, no alert) and an incomplete split (`under_claim`, plain mismatch, no over-claim alert). It reads the NF total from the persisted `attachment_analyses`, so run the `analyze-docs` agent + `apply-extractions` before `analyze`; a group with no extractable NF total is skipped. No D1 schema change — the alert flows through the existing `alerts` table/view.
- **Self-improving classification loop (feature 007):** a thin orchestrator (`improve-classification` skill) loops **analyze → review → fix → re-queue affected + re-run** to drive down _system's-fault_ mismatches while preserving _real_ findings. Vision/judgment/codegen stay in context-isolated workers (`analyze-docs`, `review-mismatch`, `fix-mismatch` agents); all bookkeeping is deterministic Python in `scripts/analysis/verdicts.py` exposed as two CLI commands — `record-verdict` (persist one verdict) and `loop-state` (recompute the open set, findings, `affected_attachment_ids`, and the `terminate` signal: `converged` / `max-iterations` / `no-progress`). Between iterations the loop re-queues only `affected_attachment_ids` by marking them pending in D1 (`mark-pending`, which clears `classified_at`) and re-runs the period — scope is DB-controlled, not threaded as id flags (feature 016). Verdicts + loop state live in `data/scrape/<period>.verdicts.json` (the **only** writer is the CLI; agents never edit it) — no D1 schema. A **mismatch identity** key (`period|kind|attachment_id|entry_id`, or `period|kind|sorted(attachment_ids)` for `duplicate_billing`) excludes volatile extracted values so re-reads don't look like new mismatches. The `mismatches` summary now carries `page_refs` (`{attachment_id, page_label, read_path}`) so the review worker opens evidence directly (FR-004). Fixes are **human-gated**: the fix worker opens a PR, the loop never merges.

## Agents

- **PM agent** (`.claude/agents/pm.md`, feature `001-repo-self-assessment`): a standalone,
  separate-context subagent that inventories repository capabilities, measures them against the
  fraud/forgery-detection goal (`docs/SCOPE-fraud-detection.md`), and writes a dated assessment
  report to `docs/assessments/` plus a ranked next-feature recommendation. On acceptance it drops
  a self-contained feature description into `specs/_handoff/` for a separate speckit-running agent
  to pick up. Advisory only — it never modifies app code, schema, or data.
- **analyze-docs agent** (`.claude/agents/analyze-docs.md`, feature `006-analyze-docs-agent`): the
  context-isolated **vision/analysis step**. Given a period it runs the whole classification analysis
  over that period's _pending_ attachments (`classified_at IS NULL`; scope is DB-controlled — re-runs
  are set up with `mark-pending`, not id flags) and returns ONLY a terse mismatch summary, so a
  caller (e.g. an orchestrator) stays context-clean. It does **not** read page images or run
  `docs-plan` itself — it invokes the **`classify-period`** skill (which owns `docs-plan` + the
  per-page `classify-doc-page` fan-out), then runs
  `apply-extractions` → `analyze` → `mismatches` and hands back that JSON. Tools: `Bash, Skill, Read,
Glob`. Deciding true-vs-false on a mismatch, and any fix, are the **review** and **fix** steps below
  (feature `007-classification-improve-loop`).
- **review-mismatch agent** (`.claude/agents/review-mismatch.md`, feature
  `007-classification-improve-loop`): the context-isolated **review step**. Given ONE mismatch (from
  the `mismatches` summary, which now carries `page_refs`) it opens the page image(s) + ledger entry
  and returns a terse **verdict** — `true` (a real finding, surfaced/never fixed), `false` (a system
  bug, with a root-cause hypothesis), `transient` (an incidental misread), or `page-error`. It writes
  nothing; the orchestrator persists the verdict via `record-verdict`. Tools: `Read, Glob, Bash`.
- **fix-mismatch agent** (`.claude/agents/fix-mismatch.md`, feature
  `007-classification-improve-loop`): the context-isolated **fix step**. Given one `false` mismatch +
  its root cause, it runs the `speckit` workflow on a dedicated branch and **opens a PR — never
  merges** (fixes are human-gated). Returns only a terse result (PR ref + one-line summary). Tools:
  `Bash, Read, Edit, Write, Skill, Glob, Grep`.
- **improve-classification skill** (`.claude/skills/improve-classification/SKILL.md`, feature
  `007-classification-improve-loop`): the thin **orchestrator** for the self-improving loop. It runs
  in the **main** context (so it can spawn the workers — a subagent cannot) and loops `analyze-docs`
  → review each mismatch → fix each `false` (human-gated PR) → **`mark-pending` the affected attachments
  in D1 + re-run the period** → until `loop-state` reports `terminate` (`converged` / `max-iterations` /
  `no-progress`). It holds
  no heavy state: all bookkeeping lives in `<period>.verdicts.json` via the `loop-state` /
  `record-verdict` CLI.
