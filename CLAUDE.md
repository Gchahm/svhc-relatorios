# CLAUDE.md

## Project Overview

Next.js 15 app deployed to Cloudflare Workers via OpenNext. Fiscal auditing tool for condominium SVHC.

## Tech Stack

- **Framework:** Next.js 15 (App Router, React 19)
- **Deployment:** Cloudflare Workers via `@opennextjs/cloudflare`
- **Database:** Cloudflare D1 (SQLite) with Drizzle ORM
- **Auth:** better-auth with Cloudflare adapter (openAPI plugin)
- **UI:** shadcn/ui (new-york style), Tailwind CSS 3, Radix primitives, lucide-react icons
- **Package manager:** pnpm
- **Formatting:** Prettier (4-space tabs, 120 print width, double quotes, trailing commas es5)

## Key Commands

```bash
pnpm dev              # Local dev server
pnpm build            # Next.js build
pnpm deploy           # Build + deploy to Cloudflare
pnpm preview          # Build + local Cloudflare preview

# Database
pnpm db:generate      # Generate Drizzle migrations
pnpm db:migrate:dev   # Apply migrations locally
pnpm db:migrate:prod  # Apply migrations to production
pnpm db:studio:dev    # Drizzle Studio (local)

# Auth schema
pnpm auth:update      # Regenerate auth schema + format

# Formatting

```

## Important Patterns

- **DB access:** Always use `await getDb()` from `@/db` - it retrieves the D1 binding from Cloudflare context
- **Auth schema:** `src/db/auth.schema.ts` is auto-generated. Run `pnpm auth:update` after changing auth config
- **Cloudflare context:** `getCloudflareContext()` is only available at runtime, not during build. The auth module has a separate static config for CLI schema generation
- **Middleware:** Checks auth by fetching `/api/auth/get-session` internally to avoid Edge Runtime eval issues
- **Path alias:** `@/*` maps to `./src/*`
- **Environment:** Cloudflare bindings defined in `wrangler.toml`: `DATABASE` (D1), `KV`, `DOCUMENTS` (R2), `ASSETS`; env vars: `BETTER_AUTH_URL`, `BETTER_AUTH_TRUSTED_ORIGINS`
- **Attachments vs. Documents (terminology тАФ feature 015; entity built in feature 020):** An **attachment** (table `attachments`, one per entry тАФ the renamed former `documents`) is the multi-page bundle downloaded from the brcondos portal; its analysis chain is `attachment_analyses` (roll-up, FK `attachment_id`) + `attachment_analysis_records` (per-page, FK `attachment_analysis_id`). A **document** is a _real_ fiscal document (an NF/NFS-e/receipt/boleto identified within attachment pages) and is now a **built** entity (table `documents`, **N:N with entries** via the join table `document_entries`). A document is **global** (not period-scoped) and identified by the semantic key **(normalized `document_number`, 14-digit `issuer_cnpj`)** тАФ uniquely indexed, so the same invoice re-scanned/re-submitted in another period resolves to one row; `total_value` is the invoice gross (max confident `nf_total_for_reconciliation` across its analyses). The pipeline builds it from `attachment_analyses` (see **Documents entity & overpayment** below). KEEP exceptions where "document" is intentional: the portal id `external_document_id`, the real-doc fields `document_type`/`document_number` and VLM `tipo_documento`, the R2 binding `DOCUMENTS`/bucket `fiscal-documents`/`getDocumentsBucket()` (storage infra), the portal `/ViewDocuments/` URL + scraper `extractors/documentos.py`, and the `--download-docs` flag.
- **Documents entity & overpayment (feature 020):** `scripts/analysis/documents.py` owns the document entity. `build_documents(target)` reads ALL `attachment_analyses` from D1 (global тАФ join `attachment_analyses тЖТ attachments тЖТ entries`, independent of any `--periodo` filter), and for each analysis confidently carrying BOTH a normalized number AND a 14-digit CNPJ, upserts one `documents` row per unique key (deterministic id `det_id("document", number, cnpj)`; `total_value` = max confident reconciliation total) and one `document_entries` link per referencing entry (`det_id("document_entry", doc_id, entry_id)`, with `source_attachment_id` for provenance; the claimed amount is read LIVE from `entries.amount`, never frozen). Analyses missing either field create nothing. The write is **authoritative** (feature 025 / issue #36): `documents`/`document_entries` are pure derived tables with no user-owned state, so any persisted document/link whose id the current analyses no longer produce is **pruned** (`DELETE тАж WHERE id NOT IN (<desired ids>)`, or unconditional `DELETE FROM` when the desired set is empty) тАФ otherwise a re-classification that changes an analysis's `(number, cnpj)` would leave a zombie document + link that distorts `/dashboard/documents` and can fire a false `document_overpayment`. The prune DELETEs and the upsert INSERTs are one **atomic** `execute_sql` batch (the feature-024 idiom: `_prune_sql(...)` + `d1.upsert_sql(...)` submitted together), so a partial failure never removes stale rows without writing the new ones; the run log reports `upserted X / pruned Y` (a pre-read of the persisted id sets gives the pruned count). It is exposed as the **`build-documents`** CLI command (global, no `--periodo`) and is also invoked at the start of `run_analysis` (after load, before the checks), so `analyze` (and the `analyze-docs` agent's applyтЖТanalyze sequence) produces documents automatically. `check_document_overpayment(target)` raises a **`document_overpayment`** (critical) alert when a document's linked entries' live amounts sum above `total_value` beyond the `reconcile_group` tolerance; its `reference_period` is the max linked period and its `metadata` (`{document_id, document_number, issuer_cnpj, total_value, sum_entries, over_amount, entry_ids}`) drives the feature-018 per-entry deep links. The writeback is a global, **atomic** `DELETE FROM alerts WHERE type='document_overpayment'` + insert in one `run_analysis` batch (feature 024; idempotent regardless of the period filter). This **supersedes** the retired `check_duplicate_billing` over-claim signal (removed from `run_advanced`); the split `amount_match` reconciliation in `apply-extractions` is a separate concern and is retained. UI: **`/dashboard/documents`** (page + `DocumentsClient`) lists/filters (type)/searches (number, issuer) documents with an over/within/under/unknown status badge and a detail dialog whose linked entries deep-link to `/dashboard/entries?period=&entry=`; backed by `GET /api/documents` (+ `/[id]`). Status math is shared in `src/lib/documents.ts` (mirrors the Python tolerance).
- **Attachment page images (R2):** Page images live in the `DOCUMENTS` R2 bucket (`fiscal-documents`), reached via `getDocumentsBucket()` in `@/lib/r2` (mirrors `getDb()`). The object key is the path **relative to `data/scrape/`** тАФ `<period>/<basename>` (e.g. `2025-12/<entry_id>_p1.png`) тАФ derived from `attachments.file_path` by `objectKeyFromFilePath()`; there is **no D1 column** for it. Page images are uploaded to R2 **during the scrape run** (`python -m scraper scrape --download-docs [--remote]`, via `getDocumentsBucket`-equivalent `wrangler` calls in `scripts/common/d1.py`); the scraper writes `attachments.file_path` as the `;`-joined `<period>/<basename>` keys it uploaded (keep that form in sync with `@/lib/r2`). For analysis, those images are materialized back from R2 into the ephemeral local cache `.cache/analysis/<period>/` (`scripts/analysis/images.py`) so the vision skill (and the legacy NF content-hash fallback/backfill) have local files. The detail dialog (`AttachmentAnalysisDetailDialog`) renders each page via `PageImageViewer` next to its extracted record; the auth-gated route `GET /api/attachment-analyses/[id]/image/[page]` resolves the key server-side (the client never sends a raw key) and streams the bytes, while `GET /api/attachment-analyses/[id]/pages` lists the pages. Locally Miniflare fabricates the bucket; in prod run `wrangler r2 bucket create fiscal-documents` once.
- **Pre-commit:** Always run `pnpm lint` and `pnpm format` before committing to ensure code passes ESLint and Prettier
- **Direct-to-D1 scrape & analysis (feature 014):** the scraper and analysis pipeline read/write **Cloudflare D1 + R2 directly** тАФ there is no `data/scrape/<period>.json` and no separate import step. `python -m scraper scrape --download-docs [--remote]` upserts a period's ledger rows into D1 and uploads its page images to R2 in one run (default **local**; `--remote` for production). The analysis CLI (`python -m analysis тАж`) takes the same `--remote` flag, loads periods from D1 (`scripts/analysis/loader.py`), and materializes images from R2 into `.cache/analysis/` (`scripts/analysis/images.py`). All D1/R2 access from Python goes through `scripts/common/d1.py` (a thin `wrangler`-CLI wrapper: `query`/`upsert_tables`/`upsert_sql`/`execute_sql`/`put_object`/`get_object`), which also owns the SQL generation (escaping, `TABLE_ORDER`, `INSERT OR REPLACE`, `analysis_records` flattening) formerly in the retired `import-to-d1.mjs`/`upload-images-to-r2.mjs`.
- **Atomic delete-then-insert writebacks (feature 024 / issue #37):** the analysis "clear old rows + insert new rows" writebacks are **one D1 batch** (one implicit transaction), never two `wrangler` calls, so a partial failure can't commit the delete without the insert. `_merge_and_write` (`scripts/analysis/attachments.py`) folds the two `DELETE`s, the `attachment_analyses` `INSERT`, **and** the `attachment_state` classified-stamp upsert (feature 026 / issue #33 тАФ the stamp moved off the mirror table `attachments` into the analysis-owned `attachment_state` table) into a single `execute_sql` тАФ so a failed insert leaves the attachment **pending** (self-healing on re-run), never stamped-classified-but-empty. The per-period and global `document_overpayment` alert writebacks in `run_analysis` (`scripts/analysis/__init__.py`) likewise `DELETE тАж + INSERT тАж` in one `execute_sql` (DELETE-only when the new set is empty), so a failure never leaves a scope falsely empty; the resolution/notes graft (feature 023) still runs before the batch. The seam is `d1.upsert_sql(data)` тЖТ returns the `INSERT OR REPLACE` SQL **without executing** (empty string when no rows) so callers prepend their `DELETE`/`UPDATE` and submit one batch; `upsert_tables` is `upsert_sql` + `execute_sql`. Stdlib `unittest` tests live in `scripts/tests/` (run `python -m unittest discover -s scripts/tests -t scripts`).
- **Mirror-table invariant & `attachment_state` (feature 026 / issue #33):** `entries`, `attachments`, and `accountability_reports` are an **exact mirror** of brcondos тАФ **only the scraper writes them**; everything the pipeline derives lives in analysis-owned tables. The analysis-owned classification state (`classified_at`) lives in **`attachment_state`** (`attachment_id` PK тЖТ `attachments.id`, `classified_at` ms; Drizzle `attachmentState` in `src/db/fiscal.schema.ts`; migration `0012` creates it, copies the old `attachments.classified_at`, drops that column). **Pending = no `attachment_state` row OR `classified_at IS NULL`.** The pipeline reads it via the loader's `attachments LEFT JOIN attachment_state` (aliasing `classified_at`), stamps it in `_merge_and_write`'s atomic batch, and clears it in `mark-pending` (`UPDATE attachment_state SET classified_at = NULL тАж`; an entry-id scope resolves to attachment ids via a read-only subquery against `attachments`). `content_hash` **stays on `attachments`** as scraper-owned (written at scrape time + by the scraper's `run_download_docs` backfill) тАФ the analysis-side `content_hash` backfill was **removed** (`scripts/analysis/images.py` no longer writes D1; it computes the hash **in memory** for a NULL row's grouping that run only). Net: the analysis pipeline issues **zero** writes to `attachments`, so it can be diffed against a fresh scrape for forgery detection.
- **Re-scrape preserves scraper-owned linkage columns (feature 027 / issue #32):** the scraper upserts attachments via `INSERT OR REPLACE` (delete-then-insert), so a re-scrape that omits or fails `--download-docs` would reset the scraper-owned `attachments.file_path` (R2-key linkage) and `content_hash` (shared-NF grouping key) to NULL тАФ BUG-001. (`classified_at` clobbering was already fixed by feature 026 moving it to `attachment_state`.) Before building a period's attachment rows for upsert, `_scrape_periodo` (`scripts/scraper/runner.py`) reads the existing `id тЖТ {file_path, content_hash}` map for that period from D1 and runs the pure merge `preserve_existing_attachment_cols` (`scripts/scraper/preserve.py`, stdlib-only so it imports without `playwright` and is unit-tested in `scripts/tests/test_attachment_preserve.py`): for each row, a **falsy** fresh value inherits the existing non-NULL value, while a **non-NULL** fresh value (a successful in-run download) wins тАФ decided **independently per attachment id and per column**, so a partial download preserves the un-downloaded siblings. First-scrape behavior is unchanged (empty map тЗТ no-op). The shared `INSERT OR REPLACE` generator in `scripts/common/d1.py` is deliberately left untouched (no other table's write semantics change).
- **Authoritative re-scrape reconciles portal deletions (feature 028 / issue #35):** the scrape was add/update-only, so a row **removed** from the brcondos portal lingered in D1 forever тАФ mirror drift (BUG-004). After a period's upsert **succeeds**, `_reconcile_period` (`scripts/scraper/runner.py`) reads the period's current mirror id sets back from D1 and the pure, playwright-free `build_reconciliation` (`scripts/scraper/reconcile.py`, unit-tested in `scripts/tests/test_reconcile.py`) diffs them against what the scrape produced: any `entries`/`attachments`/`category_subtotals`/`approvers` row present in D1 but **absent** from the fresh set is **hard-deleted** (the mirror stays an EXACT portal mirror тАФ no soft-delete column), its analysis-owned dependents (`attachment_state`, `attachment_analyses` (+ `attachment_analysis_records`), `page_classifications`, `document_entries`) are **cascade-cleaned** (all FKs are `ON DELETE no action`, so the cascade is explicit, ordered childтЖТparent under `PRAGMA defer_foreign_keys = ON`), and one idempotent **`portal_row_vanished`** (`critical`) alert per period records the vanished rows' **frozen values** as evidence (deletions are a fraud signal, so the evidence lives in the analysis-owned `alerts.metadata`; `metadata.entry_ids` drives the feature-018 deep links). It is **one atomic D1 batch** (cascade DELETEs + alert `DELETE`-then-`INSERT`, feature-024 convention) тАФ a partial failure leaves the mirror unchanged. An attachment is stale if it's gone from the scrape **or** its owning entry vanished (entry-gone wins). Reconciliation runs **only** on the scrape-success path (a failed/retried period never reconciles тАФ an empty scrape must not be read as "portal deleted everything") and only for **explicitly re-scraped** periods (the default run skips already-present periods). The period anchor (`accountability_reports`/`scrape_runs`) and global reference tables (`vendors`/`units`/`categories`/`subcategories`) are NOT reconciled; pruning an orphaned global `documents` row is left to the feature-025 `build-documents` rebuild.
- **Scrape-time consistency validation (feature 029 / issue #39):** the scraper extracts three independent views of the same money тАФ individual `entries`, the portal's consolidated per-subcategory `category_subtotals`, and the demonstrativo headline totals (`total_receitas`/`total_despesas`) тАФ and wrote all three to D1 with no cross-check (IMP-002). After a period's upsert + reconciliation **succeed**, `_check_consistency` (`scripts/scraper/runner.py`) cross-checks them via the pure, playwright-free, analysis-free `build_consistency_writeback` (`scripts/scraper/consistency.py`, unit-tested in `scripts/tests/test_scrape_consistency.py`): (a) per-`(subcategory_id, movement_type)` entry-amount sums vs the recorded subtotals (a key present on only one side compares against `0`, catching a dropped/phantom row; C and D in one subcategory are distinct keys so they never net) and (b) the subtotal sums grouped by movement type тАФ `C` vs `total_receitas`, `D` vs `total_despesas`. Any disagreement beyond the **reused** reconciliation tolerance (тЙдR$0.05 abs OR <5% rel тАФ `within_tolerance` mirrors `nf_groups`, copied not imported since the scraper must not depend on the analysis package) raises one idempotent, period-scoped **`scrape_inconsistency`** (`warning`) alert (a ledger that doesn't add up is itself a finding тАФ a scraper bug OR tampered HTML; `metadata` carries `entry_ids` for feature-018 deep links + a `checks` evidence list). The writeback is **one atomic D1 batch** тАФ always a clearing `DELETE FROM alerts WHERE id=<stable per-period det_id>` (so a now-consistent re-scrape clears the stale alert), plus an `INSERT OR REPLACE` only when inconsistent (feature-024 convention); the user's prior resolution/notes are **grafted** onto the re-emitted alert (feature-023). It runs **only** on the scrape-success path (a failed/empty period never falsely fires). A mismatch also logs a `logger.warning` and is appended to the run's `scrape_runs.errors` field via a **separate** `consistency_notes` accumulator that does NOT flip the run `status` to `error` (a mismatch is not a fatal scrape error). **No D1 schema change** (reuses the `alerts` table + dashboard, which render any type generically), no migration, no mirror-table write, no new dependency.
- **Raw portal provenance + fail-the-row parser (feature 030 / issue #38 / IMP-001):** the mirror stored only _derived_ values тАФ `parse_brl` (`scripts/scraper/extractors/demonstrativo.py`) turned a portal amount string into a float and discarded the original, and descriptions were whitespace-normalized before storage тАФ so the raw portal text was unrecoverable from D1 (a gap in the "exactly what the portal said" evidence chain), and a single malformed cell raised and aborted the whole period. Now `entries` carries two **scraper-owned, nullable** provenance columns тАФ **`raw_amount`** (verbatim amount cell text, before parsing, e.g. `"R$ 1.234,56"`) and **`raw_description`** (verbatim description cell, before `_normalize_whitespace`/`_strip_fornecedor_prefix`) тАФ populated in `_scrape_periodo`'s entry build (the lancamentos extractor carries `valor_raw`/`descricao_raw` through; `upsert_sql` picks the keys up automatically). Analysis **never** reads/writes them (mirror invariant, feature 026); existing rows read NULL until re-scraped. The derived `amount`/`description` and all deterministic ids are unchanged; round-trip invariant: `parse_brl(raw_amount) == amount` for every stored row. `parse_brl` is now **tolerant + pure** (`float | None`, rejects empty/junk/NaN/inf via `math.isfinite`, never raises): a malformed **ledger** amount fails only **that row** (the row is skipped, a warning quotes the raw text, and a non-fatal note is appended to the run's `scrape_runs.errors`/notes channel via the IMP-002 `consistency_notes` accumulator тАФ it does NOT flip `status` to `error`), while a malformed **demonstrativo** summary value (the 5 required headline totals) stays **fatal** for the period (preserves the existing "Missing financial data" abort). Migration `0013` adds the two columns; the pure parser is unit-tested in `scripts/tests/test_parse_brl.py`. Integer-centavos storage (issue suggestion #1) and `scrape_run_id` on mirror rows (#4) are deferred to follow-ups.
- **Order-independent entry-id derivation (feature 034 / issue #40 / IMP-003):** an entry's id is `det_id("entry", period, date, description, str(amount), subcategory_id, discriminator)` тАФ the first six parts are the _natural key_, the `discriminator` disambiguates two ledger lines sharing that key (split charges). The discriminator used to be the **order-dependent occurrence index** (a 1-based counter assigned in portal-HTML row order via `entry_key_counts`), so if the portal ever reordered two duplicate rows between scrapes the entries swapped ids and everything keyed by them (attachments, `document_entries`, alert `metadata.entry_ids`, deep links) silently re-pointed to the sibling. Now the pure, stdlib-only, playwright-free **`scripts/scraper/entry_ids.py`** (mirrors `preserve.py`/`reconcile.py`/`consistency.py`; unit-tested in `scripts/tests/test_entry_ids.py`) owns the derivation: `assign_entry_ids(period, rows)` partitions by natural key, and for a **duplicate** group prefers an **order-independent portal-native discriminator** `"doc:<sorted,deduped documento_ids>"` (the `/ViewDocuments/` ids already on each lancamento) over the index; a **singleton** natural key keeps the legacy discriminator `"1"` so its id is **byte-identical** (no churn тАФ only the rare duplicate-doc ids re-mint, self-healed by the feature-028 authoritative re-scrape reconciliation). When a duplicate group has **no docs or a shared doc set** it falls back to `base#<n>`/`<n>` and records one queryable `fallback_note` (FR-005). `_scrape_periodo` builds `EntryKeyInput`s in pass 1, calls `assign_entry_ids` **once** (the index can't be computed inline тАФ it needs the whole set), then builds entry/attachment rows in pass 2; the fallback notes flow through the existing IMP-002 `consistency_notes` channel. `detect_id_drift` + the runner's `_check_id_drift` read the period's **pre-upsert** entry ids/natural keys from D1 and surface a non-fatal run note when a natural key's id moved (FR-006). No D1 schema/migration, no mirror write beyond the existing `entries` upsert, no new dependency.
- **Self-cleaning `page_classifications` staging table (feature 035 / issue #42 / IMP-005):** the staging table between the vision skill (`record-classification`) and `apply-extractions` used `INSERT OR REPLACE` but **never deleted** rows, so it grew unbounded and could hold **stale** rows (a page that vanished from an attachment's `file_path` after a re-scrape) that a future `apply-extractions` might silently re-apply. Now the table is self-cleaning via a pure helper `_prune_page_classifications_sql(attachment_ids)` in `scripts/analysis/page_classifications.py` (returns a scoped `DELETE тАж WHERE attachment_id IN (тАж);` or `""` тАФ the `upsert_sql`/`documents._prune_sql` "return SQL, caller batches it" seam): (1) **apply consumes** тАФ `_merge_and_write` (`scripts/analysis/attachments.py`) appends the staging DELETE **after** the `attachment_analyses` INSERT in its **same atomic `execute_sql` batch** (feature 024), so once the authoritative roll-up lands the now-obsolete per-page rows are dropped; a failed write rolls back the whole batch, leaving the staging rows intact and the attachment pending (self-heals). The DELETE is scoped strictly to the attachment id (a shared-NF sibling, which owns no staging rows, no-ops). (2) **`mark-pending` invalidates** тАФ `mark_pending` (`scripts/analysis/extractions.py`) folds a staging DELETE (reusing the UPDATE's exact id scope, including the read-only `attachment_id IN (SELECT id FROM attachments WHERE entry_id IN (тАж))` subquery for entry-id scope тАФ no mirror write) into the same batch that clears `attachment_state.classified_at`, so a re-queued attachment reclassifies from a clean slate. No global orphan-sweep command is added (the two hooks plus the feature-028 portal-deletion cascade, which already cascade-cleans `page_classifications`, cover all sources). No D1 schema/migration, no mirror write; unit-tested in `scripts/tests/test_prune_staging.py`.
- **Attachment analysis (Claude vision skills):** the VLM (`mlx_vlm`) flow is retired. The flow is: (1) the **`classify-period` skill** runs `docs-plan` itself for a period, which **derives the work plan from D1 and prints it as JSON to stdout** (no manifest file тАФ the shared-NF grouping comes from `attachments.content_hash`; feature 016 тАФ and each plan page carries a `recorded` flag, true when its extraction is already in D1; feature 017), then fans each representative page out to (2) the **`classify-doc-page` skill**, which views one page image (a materialized cache file) and **records that page's frozen-field extraction directly to D1** via `python -m analysis record-classification --attachment-id <id> --page <label> [--remote]` (one row per page in the `page_classifications` staging table тАФ there is **no** `.classify.json` file; feature 017). The skill is passed each page's `attachment_id` + `page_label` by the orchestrator (page-image files are named by _entry_, not attachment); the `record-classification` CLI is the contract-validation point (the former PostToolUse file-write hook is gone). (3) `python -m analysis apply-extractions --periodo <p> [--remote]` reads each page's recorded extraction **from D1** (via `D1ExtractionProvider`, sourced from the loader's `page_classifications` rows) and writes the rolled-up authoritative `attachment_analyses` (+ flattened `attachment_analysis_records`) to D1 via an **atomic** delete+insert+`attachment_state`-stamp batch (feature 024 тАФ the stamp commits with the insert, so a failed write leaves the attachment pending; feature 026 / issue #33 тАФ the stamp lives in the analysis-owned `attachment_state` table, NOT on the mirror `attachments`) тАФ so clearing the cache between classify and apply no longer loses vision work. **apply reads no page-image bytes** (extractions from D1, page labels parsed from `file_path`, grouping prefers the `attachments.content_hash` column), so it materializes from R2 **only** to hash rows whose `content_hash` is NULL for **in-memory** grouping (`attachments_needing_hash_backfill` in `scripts/analysis/images.py` guards the call тАФ analysis never persists `content_hash`; that column is scraper-owned, feature 026); when all page-bearing attachments are keyed it makes **no R2 reads** (feature 019 / issue #27). `docs-plan` and `mismatches` still materialize their images; (3b) **work selection is DB-controlled** тАФ the plan is the _pending_ set (no `attachment_state` row or its `classified_at IS NULL`, supplied by the loader's LEFT JOIN); to (re)classify a subset, mark it pending in D1 with `python -m analysis mark-pending --attachment-id <idsтАж>` (clears `attachment_state.classified_at`) rather than threading id flags through the CLI/skills (feature 016); (4) `analyze` writes alerts to D1 (atomic delete+insert per `reference_period`, feature 024) and `mismatches` prints the terse summary. The **`analyze-docs` agent** is the context-isolated wrapper that invokes `classify-period` then runs apply/analyze/mismatches and returns only the summary. Each page yields a `page_extraction` record nested as `analysis_records` under its `attachment_analyses` object; `scripts/common/d1.py:upsert_tables` flattens these into `attachment_analysis_records` on write. The `attachment_analyses` row is a heterogeneity-aware roll-up (amount validated against paid/net when a payment artifact is present); per-artifact values (gross/net/paid) live in the per-page records. The extraction-source seam is `attachments.build_attachment_analysis(..., provider)` (provider keyed by `(attachment_id, page_label)`); the staging-table seam (validate / record / read) is `scripts/analysis/page_classifications.py`; the deterministic plan/apply/summary live in `scripts/analysis/extractions.py`. The legacy `attachment_analyses.raw_response` / `attachment_analysis_records.raw_text` columns were dropped (feature 017; migration `0010`) тАФ they had been unused since the VLM path was retired.
- **Shared-NF grouping & reconciliation:** The source system attaches one Nota Fiscal to several entries (line-item splits, or principal vs. `JUROS/MULTAS`). Attachments are grouped by **byte-identical page content** via the persisted `attachments.content_hash` (`scripts/analysis/nf_groups.py:group_attachments` reads that column, written at scrape time from the page bytes by `scripts/common/hashing.py:content_hash`, with a compute-from-cache fallback for rows captured before the column existed (the durable backfill is the scraper's `run_download_docs`; analysis computes the hash in memory only тАФ feature 026 / issue #33) тАФ `file_path` and `external_document_id` differ per sibling, and the extracted NF number is noisy, so the content hash is the authoritative key). For a multi-entry group, the extraction runs **once** per unique NF (the DB-derived plan lists only the representative's pages, `classify-doc-page` views them, and `apply-extractions` fans the extraction out to siblings), then validates `amount_match` by reconciling **`sum(sibling amounts)` against the NF gross total** (`nf_total_for_reconciliation` prefers the invoice `valor_total`, else the roll-up), not the full total against each fractional entry тАФ this removes the false mismatches on splits. Single-entry attachments keep the original per-entry comparison. Tolerance is reused (`nf_groups.reconcile_group`: within 5% relative OR R$ 0.05 absolute тЖТ `reconciled`; sum > total тЖТ `over_claim`; sum < total тЖТ `under_claim`).
- **Duplicate-billing alert:** `check_duplicate_billing` (`scripts/analysis/checks/advanced.py`, run by `analyze`/`run_all_checks`) emits a `critical` alert of type `duplicate_billing` when a shared NF's sibling entries sum to **more** than the NF total (an over-claim тАФ the same NF claimed above its face value), distinct from a legitimate split (`reconciled`, no alert) and an incomplete split (`under_claim`, plain mismatch, no over-claim alert). It reads the NF total from the persisted `attachment_analyses`, so run the `analyze-docs` agent + `apply-extractions` before `analyze`; a group with no extractable NF total is skipped. No D1 schema change тАФ the alert flows through the existing `alerts` table/view.
- **Actionable alerts тАФ per-attachment mismatch alerts + deep links (feature 018):** `check_attachment_mismatches` (`scripts/analysis/checks/attachments.py`, run by `analyze`/`run_all_checks` via `run_advanced`) turns the per-attachment amount/vendor/date mismatches and page-errors тАФ previously only visible through the `mismatches` CLI тАФ into user-facing alerts: one per (attachment, kind), types `attachment_amount_mismatch` / `attachment_vendor_mismatch` / `attachment_date_mismatch` (`warning`) and `attachment_page_error` (`info`), `discriminator = attachment_id` (so ids are stable and re-runs idempotent). Detection is shared with `summarize_mismatches` via `scripts/analysis/mismatches.py:detect_attachment_mismatches` (the **single source of truth** тАФ the alert check and the `mismatches` CLI can't drift; reconciliation is already baked into the persisted `attachment_analyses` match flags it reads). Each alert's `metadata` carries `{attachment_id, entry_id, kind, ledger_value, extracted_value}`. **UI side:** `/api/alerts` now selects `metadata`; `AlertsClient` parses it (`affectedEntryIds` = `entry_ids` ?? `[entry_id]`) and renders an affected-entry **deep link** per entry (a `Popover` list when an alert spans several, e.g. `duplicate_billing`/`duplicate_entry`/`negative_credit`/`large_expense_no_attachment`). The link is `/dashboard/entries?period=<YYYY-MM>&entry=<entryId>`; `EntriesClient` reads those params (under a `Suspense` boundary), selects the period, scrolls+highlights the row, and auto-opens the `AttachmentAnalysisDetailDialog` for that entry via the existing `analysisByEntry` map (no dialog when the entry has no analysis тАФ no error). No D1 schema change (`alerts.metadata` already existed). Alert `resolved` state is independent of the loop's `verdicts.json`.
- **Self-improving classification loop (feature 007):** a thin orchestrator (`improve-classification` skill) loops **analyze тЖТ review тЖТ fix тЖТ re-queue affected + re-run** to drive down _system's-fault_ mismatches while preserving _real_ findings. Vision/judgment/codegen stay in context-isolated workers (`analyze-docs`, `review-mismatch`, `fix-mismatch` agents); all bookkeeping is deterministic Python in `scripts/analysis/verdicts.py` exposed as two CLI commands тАФ `record-verdict` (persist one verdict) and `loop-state` (recompute the open set, findings, `affected_attachment_ids`, and the `terminate` signal: `converged` / `max-iterations` / `no-progress`). Between iterations the loop re-queues only `affected_attachment_ids` by marking them pending in D1 (`mark-pending`, which clears `classified_at`) and re-runs the period тАФ scope is DB-controlled, not threaded as id flags (feature 016). Verdicts + loop state live in `data/scrape/<period>.verdicts.json` (the **only** writer is the CLI; agents never edit it) тАФ no D1 schema. A **mismatch identity** key (`period|kind|attachment_id|entry_id`, or `period|kind|sorted(attachment_ids)` for `duplicate_billing`) excludes volatile extracted values so re-reads don't look like new mismatches. The `mismatches` summary now carries `page_refs` (`{attachment_id, page_label, read_path}`) so the review worker opens evidence directly (FR-004). Fixes are **human-gated**: the fix worker opens a PR, the loop never merges.

## Agents

- **PM agent** (`.claude/agents/pm.md`, feature `001-repo-self-assessment`): a standalone,
  separate-context subagent that inventories repository capabilities, measures them against the
  fraud/forgery-detection goal (`docs/SCOPE-fraud-detection.md`), and writes a dated assessment
  report to `docs/assessments/` plus a ranked next-feature recommendation. On acceptance it drops
  a self-contained feature description into `specs/_handoff/` for a separate speckit-running agent
  to pick up. Advisory only тАФ it never modifies app code, schema, or data.
- **analyze-docs agent** (`.claude/agents/analyze-docs.md`, feature `006-analyze-docs-agent`): the
  context-isolated **vision/analysis step**. Given a period it runs the whole classification analysis
  over that period's _pending_ attachments (`classified_at IS NULL`; scope is DB-controlled тАФ re-runs
  are set up with `mark-pending`, not id flags) and returns ONLY a terse mismatch summary, so a
  caller (e.g. an orchestrator) stays context-clean. It does **not** read page images or run
  `docs-plan` itself тАФ it invokes the **`classify-period`** skill (which owns `docs-plan` + the
  per-page `classify-doc-page` fan-out), then runs
  `apply-extractions` тЖТ `analyze` тЖТ `mismatches` and hands back that JSON. Tools: `Bash, Skill, Read,
Glob`. Deciding true-vs-false on a mismatch, and any fix, are the **review** and **fix** steps below
  (feature `007-classification-improve-loop`).
- **review-mismatch agent** (`.claude/agents/review-mismatch.md`, feature
  `007-classification-improve-loop`): the context-isolated **review step**. Given ONE mismatch (from
  the `mismatches` summary, which now carries `page_refs`) it opens the page image(s) + ledger entry
  and returns a terse **verdict** тАФ `true` (a real finding, surfaced/never fixed), `false` (a system
  bug, with a root-cause hypothesis), `transient` (an incidental misread), or `page-error`. It writes
  nothing; the orchestrator persists the verdict via `record-verdict`. Tools: `Read, Glob, Bash`.
- **fix-mismatch agent** (`.claude/agents/fix-mismatch.md`, feature
  `007-classification-improve-loop`): the context-isolated **fix step**. Given one `false` mismatch +
  its root cause, it runs the `speckit` workflow on a dedicated branch and **opens a PR тАФ never
  merges** (fixes are human-gated). Returns only a terse result (PR ref + one-line summary). Tools:
  `Bash, Read, Edit, Write, Skill, Glob, Grep`.
- **improve-classification skill** (`.claude/skills/improve-classification/SKILL.md`, feature
  `007-classification-improve-loop`): the thin **orchestrator** for the self-improving loop. It runs
  in the **main** context (so it can spawn the workers тАФ a subagent cannot) and loops `analyze-docs`
  тЖТ review each mismatch тЖТ fix each `false` (human-gated PR) тЖТ **`mark-pending` the affected attachments
  in D1 + re-run the period** тЖТ until `loop-state` reports `terminate` (`converged` / `max-iterations` /
  `no-progress`). It holds
  no heavy state: all bookkeeping lives in `<period>.verdicts.json` via the `loop-state` /
  `record-verdict` CLI.
